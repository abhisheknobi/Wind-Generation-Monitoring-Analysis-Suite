## 1. What Is This Project?

### The Real-World Problem

The UK generates electricity from wind turbines. Wind is unpredictable — sometimes it blows hard, sometimes it doesn't. Grid operators (the people who keep the lights on) need to know TWO things:

1. **How much wind power will we have tomorrow?** → They rely on FORECASTS
2. **How much can we DEPEND on?** → They need RELIABILITY analysis

But forecasts are never perfect. So there's a third question:

3. **How wrong are the forecasts, and can we find patterns in the mistakes?**

This project answers all three questions using real UK energy data.

### What This Project Actually Does

| Component | What It Does | Analogy |
|-----------|-------------|---------|
| **Web App** (React + Node.js) | Shows a chart comparing forecast vs actual wind generation with accuracy metrics | A dashboard — like a car's speedometer showing "expected speed" vs "actual speed" |
| **Notebook 1** (Jupyter) | Analyzes forecast errors — how big, how often, when they're worst | A teacher grading exams and writing a grade report |
| **Notebook 2** (Jupyter) | Analyzes historical wind data to recommend how much wind power is reliable | An insurance actuary calculating risk — "what's the minimum we can count on?" |

## 2. The "Model" — What It Is and Isn't

### THE MOST IMPORTANT THING TO UNDERSTAND

**There is NO trained AI/ML model in this project. We did NOT build, train, or deploy any machine learning model.**

The word "model" refers to the **BMRS forecasting system** — a prediction system operated by **Elexon** (the company that runs the UK's electricity balancing mechanism). They have their own internal forecasting systems (likely numerical weather prediction models combined with statistical methods) that predict how much wind power the UK will generate.

### What We Have Access To

We have ZERO access to how their model works internally. It's a complete black box. What we CAN access (for free, via their public API):

| What | How We Get It | What It Tells Us |
|------|--------------|-----------------|
| **Their predictions** | WINDFOR dataset via API | What Elexon's model THOUGHT would happen |
| **What actually happened** | FUELHH dataset via API | What REALLY happened |

### Our Role

We are NOT the forecaster. We are the **analyst who evaluates the forecaster**.

**Analogy:** Imagine a weather channel that predicts temperatures every day. We don't build our own weather model. Instead, we:
1. Record what they predicted each day
2. Record the actual temperature each day
3. Compare them and write a report: "The weather channel was off by 3°F on average, worst in winter, better for short-term forecasts"

That's exactly what this project does — but for wind power instead of temperature.

## 3. Where the Data Comes From

### The Source: BMRS Elexon API

**URL:** `https://data.elexon.co.uk/bmrs/api/v1`

**What is BMRS?** The Balancing Mechanism Reporting Service. The UK government requires that all electricity generation and trading data be published transparently. BMRS is where this data lives.

**Why is it free?** UK energy regulation requires transparency. This is public data funded by energy market levies. No API key, no sign-up, no cost.

### Two Datasets We Use

#### Dataset 1: FUELHH (Actual Generation)

- **Endpoint:** `/datasets/FUELHH/stream`
- **Full name:** Fuel type Half-Hourly
- **What it contains:** Every 30 minutes, the actual electricity generated by EACH fuel type (wind, gas, nuclear, solar, etc.)
- **We filter for:** `fuelType = "WIND"` (we only care about wind)
- **Key fields:**
  - `startTime` — the 30-minute time slot (e.g., "2025-01-15T18:00:00Z")
  - `generation` — actual wind power in MW (megawatts)

#### Dataset 2: WINDFOR (Wind Forecasts)

- **Endpoint:** `/datasets/WINDFOR/stream`
- **Full name:** Wind Forecast
- **What it contains:** Predicted wind generation values, with timestamps of when each prediction was made
- **Key fields:**
  - `startTime` — the time being predicted FOR (target time)
  - `publishTime` — when the forecast was CREATED (published)
  - `generation` — predicted wind power in MW

#### The Critical Difference

```
FUELHH:  "At 18:00, wind generated 5,200 MW"             (FACT — already happened)
WINDFOR: "At 14:00, we predicted 18:00 would be 5,500 MW" (PREDICTION — made earlier)
```

The gap between `startTime` and `publishTime` is called the **forecast horizon** — how far in advance the prediction was made.

### How We Fetch It (Code)

In `server/services/bmrsApi.js`:
```javascript
// Fetch actual generation
const url = "https://data.elexon.co.uk/bmrs/api/v1/datasets/FUELHH/stream";
const response = await axios.get(url, { params: { from: "2025-01-01", to: "2025-01-31" } });
// Filter response for fuelType === "WIND"
```

In Python (`python-analytics/services/data_fetcher.py`):
```python
# Same API, same data, but returns a pandas DataFrame
url = "https://data.elexon.co.uk/bmrs/api/v1/datasets/FUELHH/stream"
response = requests.get(url, params={"from": from_date, "to": to_date})
df = pd.DataFrame(response.json())
df = df[df["fuelType"].str.upper() == "WIND"]
```

Both do the same thing — one in JavaScript (for the web server), one in Python (for notebooks and analytics).

### Why Three Separate Services?

| Service | Language | Why? |
|---------|----------|------|
| **Frontend** (React) | JavaScript | Browsers run JavaScript. React is the industry standard for interactive UIs. |
| **Backend** (Node.js) | JavaScript | Acts as a middleman — the browser can't call the BMRS API directly (CORS security). Also caches data. |
| **Analytics** (Python) | Python | Python has pandas/numpy/scipy — the best data analysis libraries. JavaScript doesn't have equivalents. |

**Why not call the BMRS API directly from the browser?**
- **CORS** (Cross-Origin Resource Sharing): Browsers block requests to external APIs for security. The BMRS API doesn't set CORS headers allowing browser requests.
- **Data processing**: Raw API data needs filtering and transformation. Doing this on the server keeps the frontend simple.
- **Caching**: The server caches responses so repeated requests don't hit the external API.

## 5. Backend Deep Dive (Node.js)

### File: `server/services/bmrsApi.js` — The API Client

**Purpose:** Talks to the BMRS Elexon API. Two functions:

1. `fetchActualGeneration(from, to)` → Calls FUELHH endpoint → Filters for WIND → Returns `[{ startTime, generation }]`
2. `fetchWindForecast(from, to)` → Calls WINDFOR endpoint → Returns `[{ startTime, publishTime, generation }]`

**Design decision:** 30-second timeout. The BMRS API can be slow for large date ranges (months of data). 30 seconds is generous enough to avoid false failures.

### File: `server/services/dataProcessor.js` — The Brain

This is the most important backend file. Three functions:

#### Function 1: `selectForecastsByHorizon(forecasts, horizonHours)`

**The problem it solves:** For any given time (e.g., Jan 15 at 18:00), there might be 10+ forecasts made at different times. Which one do we display?

**The algorithm:**
```
1. Group all forecasts by their target time (startTime)
2. For each target time:
   a. Calculate the cutoff = targetTime - horizonHours
      Example: 18:00 - 4 hours = 14:00
   b. Keep only forecasts published BEFORE the cutoff
      (published before 14:00 in this example)
   c. Also filter: horizon must be 0-48 hours (no forecasts from last week)
   d. From the remaining, pick the LATEST one
      (most recent forecast = most accurate, but still meets the horizon)
3. Sort results by time
```

**Why pick the LATEST valid forecast?**
- A forecast made at 13:30 (4.5h ahead) is more accurate than one made at 10:00 (8h ahead)
- Both meet the 4h horizon requirement
- We want the best allowable forecast, not just any valid one

**Why the 4-hour minimum?**
- In real energy markets, grid operators need lead time to ramp up backup generators
- A forecast made 5 minutes ago isn't useful for planning
- 4 hours is a common operational planning horizon in the UK energy market

#### Function 2: `combineData(actuals, forecasts)`

**The problem:** We have two separate arrays — actuals and forecasts — with different timestamps. The chart needs ONE array where each entry has `{ time, actual, forecast }`.

**The algorithm:**
```
1. Create a dictionary keyed by timestamp
2. Loop through actuals — add each as { time, actual: value, forecast: null }
3. Loop through forecasts — add/update each timestamp with forecast value
4. Convert dictionary to sorted array
```

**Why nulls matter:** If a timestamp has an actual but no forecast (or vice versa), the missing value is `null`. The chart component uses `connectNulls={false}`, which creates GAPS in the line instead of connecting across missing data. This is critical — the SRS says "for missing forecast data, ignore them and don't plot any values."

#### Function 3: `calculateStats(combinedData)`

Computes error metrics from the combined data. Only uses data points where BOTH actual AND forecast exist (<code>paired = data.filter(d => d.actual !== null && d.forecast !== null)</code>).

Metrics calculated:
- **MAE** = sum(|error|) / count
- **RMSE** = sqrt(sum(error²) / count)
- **MAPE** = sum(|error|/actual × 100) / count (skipping actual=0 to avoid ÷0)
- **Bias** = sum(error) / count (signed, not absolute)

### File: `server/routes/generation.js` — The API Endpoints

Three routes the frontend can call:

| Route | What It Does | When Frontend Calls It |
|-------|-------------|----------------------|
| `GET /api/generation/actual` | Returns raw actual data | Rarely — for debugging |
| `GET /api/generation/forecast` | Returns processed forecasts | Rarely — for debugging |
| `GET /api/generation/combined` | **THE MAIN ONE** — returns chart data + stats | Every time you click "Fetch Data" |

**The `/combined` route does everything:**
1. Fetches actuals AND forecasts in parallel (`Promise.all` — both API calls run simultaneously, ~2x faster than sequential)
2. Applies horizon filtering via `selectForecastsByHorizon()`
3. Merges via `combineData()`
4. Calculates stats via `calculateStats()`
5. Caches the result for 5 minutes

### Caching Strategy

```javascript
const cache = new NodeCache({ stdTTL: 300 }); // 5 minutes
```

**Why cache?**
- The BMRS API is slow (1-5 seconds per request)
- If a user clicks "Fetch Data" twice with the same params, the second click returns instantly
- Cache key = combination of endpoint + from + to + horizon — different params = different cache entries

**Why 5 minutes?**
- Trade-off: too short (30s) → too many API calls; too long (1 hour) → stale data
- 5 minutes is reasonable — actual generation data is finalized and doesn't change, forecasts update periodically but not every second

## 8. Notebook 1: Forecast Error Analysis

### The Question We're Answering

> "The BMRS wind forecast model makes predictions. How good are those predictions? Are there patterns in its mistakes?"

### Step-by-Step Logical Path

#### Step 1: Define the Analysis Period

**Choice:** January – March 2025 (3 months)

**Reasoning:**
- Too little data (1 week = ~336 points) → statistically noisy, unreliable conclusions
- Too much data (2 years) → API calls would be very slow; older data may not reflect current model performance
- 3 months ≈ 4,300 half-hourly points → more than enough for statistical significance

**Trade-off:** We sacrifice capturing full seasonal variation (3 months = only winter/spring) in exchange for API speed and recency.

**Assumption:** The forecast model's behavior in Jan-Mar 2025 is representative of its general performance. This is reasonable for a model that hasn't been significantly updated.

#### Step 2: Fetch Raw Data

We make two API calls:
- FUELHH → actual generation (what happened)
- WINDFOR → forecasts (what was predicted)

**Assumption:** The API data is complete and accurate. Since it's the official UK balancing mechanism data, this is a strong assumption.

#### Step 3: The Forecast Horizon Problem

**The problem:** For target time 18:00, there might be forecasts published at 17:30, 14:00, 12:00, 10:00, and 06:00. Which one do we evaluate?

**First-principles reasoning:**
- In real operations, a forecast made 30 minutes ago is nearly useless — you can't change your generation plan in 30 minutes
- Grid operators typically need 4+ hours of lead time to ramp up gas turbines or import power
- So evaluating a 30-minute forecast doesn't reflect real-world accuracy

**Decision:** Use minimum 4-hour horizon (configurable).

**Algorithm:**
```
For each target time:
    horizon = startTime - publishTime
    Keep forecasts where: 4 ≤ horizon ≤ 48 hours
    From valid forecasts: pick the one with the SMALLEST horizon
    (latest forecast that still meets the 4h minimum = most accurate available)
```

**Why cap at 48 hours?**
- Weather prediction degrades dramatically beyond ~48 hours
- Energy markets rarely use forecasts beyond day-ahead
- Including 5-day-ahead forecasts would inflate error metrics unfairly

#### Step 4: Merge and Compute Errors

After selecting one forecast per target time, we merge:
```
matched = actuals INNER JOIN selected_forecasts ON startTime
```

Inner join means: only keep time points where BOTH actual and forecast exist. If a forecast is missing for a time slot, we skip it entirely.

Then compute: `error = forecast - actual`
- Positive error → model predicted MORE wind than actually occurred (over-prediction)
- Negative error → model predicted LESS wind (under-prediction)

**Why this sign convention?** It tells us the DIRECTION of the mistake, not just the size. This matters for grid operations — over-prediction means you have less power than expected (potentially dangerous), while under-prediction means excess power (wasteful but safe).

#### Step 5: Summary Error Metrics

Each metric answers a different question:

##### MAE (Mean Absolute Error)

$$MAE = \frac{1}{n}\sum_{i=1}^{n}|forecast_i - actual_i|$$

**English:** "Take every error, ignore whether it's positive or negative (take absolute value), and average them."

**What it tells you:** "On average, the forecast is wrong by ___ MW."

**Strength:** Intuitive, easy to explain
**Weakness:** Treats a 100 MW error and a 10,000 MW error "equally" in the average — doesn't highlight extreme misses

##### RMSE (Root Mean Squared Error)

$$RMSE = \sqrt{\frac{1}{n}\sum_{i=1}^{n}(forecast_i - actual_i)^2}$$

**English:** "Square each error (making big errors much bigger), average them, then square-root."

**What it tells you:** "How bad are the LARGE errors?"

**Why use it alongside MAE?** If RMSE is much larger than MAE, it means there are occasional very large errors (outliers). RMSE ≈ MAE means errors are relatively uniform.

**Mathematical insight:** Squaring penalizes large errors quadratically. A 1000 MW error contributes 1,000,000 to the sum, while a 100 MW error contributes only 10,000. So RMSE is dominated by the biggest mistakes.

##### MAPE (Mean Absolute Percentage Error)

$$MAPE = \frac{1}{n}\sum_{i=1}^{n}\frac{|forecast_i - actual_i|}{actual_i} \times 100\%$$

**English:** "What percentage of actual generation is the error?"

**Why useful?** An error of 500 MW means very different things depending on context:
- If actual = 10,000 MW → 500 MW error = 5% → pretty good
- If actual = 500 MW → 500 MW error = 100% → terrible

MAPE normalizes for this.

**Caveat:** We skip data points where actual = 0 (division by zero). This is a known limitation of MAPE.

##### Median Absolute Error

**Why not just use MAE?** Because MAE is an AVERAGE, it's distorted by extreme values. If 99% of errors are 200 MW but one error is 10,000 MW, the MAE could be 300+ MW even though the "typical" error is 200 MW.

The median gives the "middle" error — 50% of errors are below this, 50% above. It's robust to outliers.

**If median << MAE:** The distribution is right-skewed — most errors are small, but a few are very large. This is valuable operational information.

##### P99 Absolute Error

**English:** "The error that 99% of forecasts are better than."

**Why it matters:** For grid operations, you need to plan for the worst case. If P99 = 5,000 MW, you need 5,000 MW of backup capacity to cover 99% of situations.

##### Bias

$$Bias = \frac{1}{n}\sum_{i=1}^{n}(forecast_i - actual_i)$$

**English:** "On average, does the model predict too high or too low?"

**Why it matters:** If bias = +200 MW, the model systematically over-predicts by 200 MW. You could apply a simple correction: `corrected_forecast = forecast - 200`. This would instantly improve average accuracy.

A bias near zero means the model doesn't systematically lean in either direction.

#### Step 6: Error Distribution Visualization

We plot TWO histograms:
1. **Signed error** (can be negative or positive) — shows bias and symmetry
2. **Absolute error** (always positive) — shows magnitude

**What to look for:**
- Is the signed distribution centered on zero? (No bias)
- Is it symmetric? (If asymmetric, the model makes different types of mistakes in each direction)
- Does the absolute error have a long right tail? (Occasional large misses)

#### Step 7: Error vs. Forecast Horizon

**Hypothesis from first principles:** Weather is a chaotic system. Tiny differences in initial conditions amplify over time (the "butterfly effect"). Therefore, predictions further into the future should be less accurate.

**How we test it:**
1. Bucket all forecasts by their horizon: 0-4h, 4-8h, 8-12h, 12-18h, 18-24h, 24-36h, 36-48h
2. Compute MAE for each bucket
3. Plot and look for an increasing trend

**Why these specific bucket sizes?**
- Short horizons (0-4h, 4-8h): Error changes rapidly at short timescales, so we use narrow buckets to capture the gradient
- Long horizons (24-36h, 36-48h): Error growth typically plateaus at longer timescales, so wider buckets are fine and give us more data per bucket for statistical stability

**Expected result:** Monotonically increasing MAE from left to right. If we see a DIP at a certain horizon, that would suggest the model has a structural issue at specific timescales — worth investigating further.

#### Step 8: Error by Time of Day

**Question:** Are forecasts worse at certain hours?

**Physical reasoning for why this could happen:**
- **Daytime:** Solar heating creates convective turbulence → wind becomes gusty and irregular → harder to predict precisely
- **Night:** Atmosphere stabilizes (radiative cooling) → smoother, more laminar wind flows → potentially easier to predict
- **Dawn/dusk:** Rapid transitions in atmospheric stability → the model might struggle with timing of these transitions
- **Wind ramp events:** Rapid increases/decreases in wind speed often happen at specific times — if the model's timing is slightly off, these show up as large errors at those hours

**How we test it:**
1. Add hour-of-day column (0-23) to each data point
2. Group by hour → compute MAE per hour
3. Plot: bar chart of MAE by hour + line chart of avg actual vs avg forecast

#### Step 9: Conclusions

We draw conclusions based on evidence, not assumptions:

1. **MAE and RMSE values** → quantify overall accuracy
2. **RMSE > MAE** → confirms heavy-tailed distribution (large outlier errors exist)
3. **Bias direction** → systematic over or under-prediction
4. **Median < MAE** → right-skewed errors (most small, some large)
5. **P99 >> MAE** → worst-case errors are much larger than typical
6. **Horizon trend** → confirms weather unpredictability grows with time
7. **Hourly pattern** → identifies vulnerable time periods for grid planning

## 9. Notebook 2: Wind Reliability Analysis

### The Question

> "How many MW of wind power can we RELIABLY count on being available?"

This is fundamentally different from Notebook 1. Notebook 1 evaluates forecasts. Notebook 2 analyzes actual historical generation to assess dependability.

### Why This Question Matters

**The grid operator's dilemma:**
- UK have ~30,000 MW of installed wind capacity (nameplate)
- Average generation is maybe ~8,000-10,000 MW (wind doesn't blow at full speed all the time)
- But on calm days, it might drop to ~500 MW
- If you PLAN to always have 10,000 MW available and actual drops to 500 MW, you have a 9,500 MW shortfall → potential blackouts

So the question isn't "how much wind power do we HAVE?" but "how much can we DEPEND on?"

### Step-by-Step Logical Path

#### Step 1: Fetch a Full Year of Data

**Choice:** All of 2025

**Reasoning:**
- Wind is seasonal (windier in winter, calmer in summer)
- Using only 3 months would miss seasonal variation
- A full year captures all seasons, holidays, and weather patterns
- ~17,500 data points per year — very statistically robust

**Limitation:** One year might not capture year-to-year variability. Some years are windier than others. Ideally, we'd use 5-10 years of data. But the BMRS instruction says "data from January 2025 onwards," and this is the data we have.

#### Step 2: Descriptive Statistics

First, understand the basics:
- **Mean** — average generation (BUT NOT a reliable planning number)
- **Median** — the middle value (50% above, 50% below)
- **Standard deviation** — how spread out the values are
- **Min/Max** — the extremes
- **Coefficient of variation (CV)** — std/mean × 100%

**The CV is the key insight here.** For stable power sources:
- Nuclear: CV ≈ 5-10% (very predictable)
- Gas: CV ≈ 10-20% (dispatched on demand)
- Wind: CV ≈ 40-60% (highly unpredictable)

A high CV mathematically proves that the mean is NOT a reliable planning number — the actual values deviate wildly from it.

#### Step 3: The Percentile Framework — Core Analytical Thinking

**Why percentiles are the RIGHT tool for this problem:**

Think of it this way. You want to answer: "What's the generation value that wind power stays ABOVE X% of the time?"

This is exactly the definition of an exceedance percentile.

**How to read it:**
- **P50 exceedance** = median = generation is above this 50% of the time
  - If you plan for this: shortfalls 50% of the time. Unacceptable.
- **P90 exceedance** = generation is above this 90% of the time
  - If you plan for this: shortfalls only 10% of the time. Manageable with backup.
- **P95 exceedance** = 95% of the time
  - More conservative, but lower MW value → expensive to implement
- **P99 exceedance** = 99% of the time
  - Almost never fails, but very low MW → barely any capacity credit for wind

**Mathematical detail:** P90 exceedance = `quantile(0.10)`. Why?
- "Exceeded 90% of the time" means "only 10% of values are BELOW this"
- The 10th percentile of the data = the value below which 10% of data falls
- Therefore: 90% of data is ABOVE the 10th percentile = P90 exceedance

#### Step 4: Duration Curve

We plot the **exceedance probability curve** (also called a duration curve):
- X-axis: MW of wind generation
- Y-axis: % of time generation exceeds that value

**How to read it:** Pick any MW value on the X-axis, go up to the curve, read across to the Y-axis. That tells you what % of the time wind generation was at or above that MW level.

This is the single most powerful chart in the analysis. Grid operators use these daily.

#### Step 5: Seasonal Patterns

**Why?** If we recommend P90 = X MW based on the full year, that includes windy winter months that inflate the number. During a calm summer month, the actual P90 might be much lower.

**Analysis:**
- Compute mean and P90 for each month separately
- Find the weakest month
- Use the weakest month's P90 as a seasonal-adjusted recommendation

**Trade-off:** Using the worst month's P90 is more conservative (lower MW) but more honestly represents year-round reliability. Using the annual P90 is higher but risky during calm periods.

#### Step 6: Time-of-Day Patterns

**Question:** Does wind generation vary consistently by hour?

**Physical reasoning:** Wind speed has diurnal (daily) patterns:
- Onshore: often stronger during the afternoon (surface heating creates wind), calmer at night
- Offshore: less diurnal variation (ocean temperature is more stable)
- National aggregate: blend of both

If there's a consistent daily pattern, it affects when we can rely on wind:
- If wind is weakest during 14:00-18:00 (peak electricity demand), that's a problem
- If wind is strongest at 02:00-06:00 (low demand), the excess is less useful

#### Step 7: Wind Drought Analysis

**The problem that percentiles DON'T capture:**

P90 tells us "90% of individual half-hours are above X MW." But it doesn't tell us about CONSECUTIVE periods. Ten scattered low half-hours are manageable. Seventy-two CONSECUTIVE hours of low wind (3 days) could be catastrophic — you'd need backup generators running continuously for 3 days.

**Algorithm:**
```
1. Define threshold = P90 exceedance value
2. Mark every half-hour as below_threshold (True/False)
3. Find consecutive runs of True values
4. Report duration, minimum generation, and dates for longest runs
```

**What this tells grid operators:** "Your backup systems must sustain output for at least N hours." This drives decisions about:
- Battery storage capacity (how many MWh)
- Gas turbine keep-warm contracts
- Interconnector agreements with neighboring countries

#### Step 8: Capacity Factor

$$Capacity Factor = \frac{Average Generation}{Maximum Generation} \times 100\%$$

**What it means:** "What fraction of its maximum potential does wind actually deliver on average?"

- UK wind CF is typically 25-35% (onshore 25-30%, offshore 35-45%)
- This reflects the fundamental intermittency of wind — turbines don't run at full speed all the time

**We use observed maximum as a proxy for installed capacity.** This is an approximation — the true installed capacity is slightly higher than the highest observed output (since ALL turbines never operate at maximum simultaneously). But it's reasonable without access to official installed capacity data.

#### Step 9: The Final Recommendation

**Our recommendation: P90 exceedance value**

**Why P90 specifically (the reasoning chain):**

1. **Why not the mean?** The mean = P50. By definition, actual generation is below the mean ~50% of the time. Planning for the mean means blackout risk in half of all time periods. This is mathematically unacceptable.

2. **Why not P95 or P99?** While more conservative:
   - P95 and P99 yield MUCH lower MW values
   - This means you'd need MUCH more expensive backup capacity
   - Grid operators already have spinning reserves and demand-side response to cover the 10% tail
   - Going beyond P90 provides diminishing returns in reliability at escalating cost

3. **Why P90?** It's the **Goldilocks zone**:
   - Reliable enough: only 10% of time periods fall short
   - Not overly conservative: doesn't waste money on unnecessary backup
   - Industry standard: all renewable energy capacity assessments worldwide use P90
   - Grid operators have tools (reserves, interconnectors, demand response) to cover the 10%

4. **Seasonal adjustment:** We also provide the worst month's P90 as a more conservative figure — ensuring year-round reliability including calm summer periods.

## 10. Key Math Concepts Explained

### Percentiles

Think of it as ranking. If you have 100 exam scores:
- The 10th percentile = the score that only 10 students scored BELOW
- The 90th percentile = the score that only 10 students scored ABOVE

For wind generation:
- quantile(0.10) = the generation level that only 10% of readings were BELOW
- This means 90% of readings were ABOVE it = P90 exceedance

### Standard Deviation

**Intuition:** "How spread out is the data?"

If mean = 5,000 MW and std = 500 MW → data is clustered tightly (± 10%)
If mean = 5,000 MW and std = 3,000 MW → data is all over the place (± 60%)

### Coefficient of Variation (CV)

$$CV = \frac{Std Dev}{Mean} \times 100\%$$

**Why not just use std dev?** Because std dev depends on scale. A std dev of 1,000 MW is tiny for nuclear (100,000 MW output) but huge for a local wind farm (2,000 MW output). CV normalizes for scale.

### Histogram

A bar chart showing how many data points fall into each range. Tells you the SHAPE of the data:
- Bell-shaped → normally distributed (most values near the center)
- Right-skewed → most values are small, but there's a long tail of large values
- Bimodal → two peaks (might indicate two different weather regimes)

### Box Plot

Shows five numbers in one picture:
```
         ┌───┐
    ─────┤   ├─────
    │    └─┬─┘    │
    │      │      │
   min    Q1 median Q3   max
              (box)
```
- Box = middle 50% of data (Q1 to Q3)
- Line in box = median
- Whiskers = extend to min/max (or 1.5×IQR)
- Dots beyond whiskers = outliers

## 11. Assumptions, Trade-offs, and Limitations

### Assumptions We Made

| Assumption | Justification | Risk If Wrong |
|-----------|---------------|---------------|
| BMRS API data is accurate | Official UK balancing mechanism — legally required to be correct | Low risk |
| 3 months enough for error analysis | >4,000 data points → statistically robust | May miss seasonal patterns |
| 1 year enough for reliability analysis | ~17,500 points, covers all seasons | May miss multi-year variability |
| Observed max ≈ installed capacity | Tight approximation; real capacity slightly higher | Capacity factor slightly underestimated |
| Forecast model is consistent over analysis period | Elexon doesn't change models frequently | If they updated the model mid-period, our analysis blends old and new performance |
| 4-hour horizon is operationally relevant | Standard in UK energy markets | Different markets may use different horizons |

### Trade-offs We Made

| Decision | Pro | Con | Why We Chose It |
|----------|-----|-----|----------------|
| P90 vs P95 vs P99 for reliability | P90 is standard, not over-conservative | 10% failure rate | Industry consensus; backup systems cover the tail |
| Annual P90 vs worst-month P90 | Annual: higher MW | Worst-month: lower but safer | Provide both; recommend worst-month for firm capacity |
| Using mean for error metrics | Simple, interpretable | Sensitive to outliers | Provide median and P99 alongside for robustness |
| 3-month analysis period | Fast API calls, recent data | Misses summer patterns | Acceptable for forecast model evaluation |
| Horizon buckets of varying width | More detail at short horizons where error changes fast | Unequal bin widths can be misleading | Labeled clearly; matches operational horizons |

### Known Limitations

1. **No demand-side analysis:** We analyze wind supply in isolation. True reliability requires comparing wind supply with electricity DEMAND. Low wind + high demand (cold winter anticyclone) is especially dangerous — our analysis doesn't capture this correlation.

2. **Static analysis:** We treat data as a static snapshot. In reality, wind capacity changes over time (new turbines installed), seasonal patterns vary year-to-year, and forecast models get updated.

3. **No geographic breakdown:** National-level aggregation smooths out local variability. A specific region might be much less reliable than the national figure suggests.

4. **No curtailment analysis:** Sometimes wind turbines are deliberately curtailed (turned off) due to grid constraints, even when wind is available. This makes actual generation appear lower than true wind availability.
